<dd><p>A DataContext represents a Great Expectations project. It is the primary entry point for a Great Expectations
deployment, with configurations and methods for all supporting components.</p>
<p>The DataContext is configured via a yml file stored in a directory called great_expectations; this configuration
file as well as managed Expectation Suites should be stored in version control. There are other ways to create a
Data Context that may be better suited for your particular deployment e.g. ephemerally or backed by GE Cloud
(coming soon). Please refer to our documentation for more details.</p>
<p>You can Validate data or generate Expectations using Execution Engines including:</p>
<blockquote>
<div><ul class="simple">
<li><p>SQL (multiple dialects supported)</p></li>
<li><p>Spark</p></li>
<li><p>Pandas</p></li>
</ul>
</div></blockquote>
<p>Your data can be stored in common locations including:</p>
<blockquote>
<div><ul class="simple">
<li><p>databases / data warehouses</p></li>
<li><p>files in s3, GCS, Azure, local storage</p></li>
<li><p>dataframes (spark and pandas) loaded into memory</p></li>
</ul>
</div></blockquote>
<p>Please see our documentation for examples on how to set up Great Expectations, connect to your data,
create Expectations, and Validate data.</p>
<p>Other configuration options you can apply to a DataContext besides how to access data include things like where to
store Expectations, Profilers, Checkpoints, Metrics, Validation Results and Data Docs and how those Stores are
configured. Take a look at our documentation for more configuration options.</p>
<p>You can create or load a DataContext from disk via the following:
<code class="docutils literal notranslate"><span class="pre">`</span>
<span class="pre">import</span> <span class="pre">great_expectations</span> <span class="pre">as</span> <span class="pre">ge</span>
<span class="pre">ge.get_context()</span>
<span class="pre">`</span></code></p>

<dd><p>Convenience method for testing yaml configs</p>
<p>test_yaml_config is a convenience method for configuring the moving
parts of a Great Expectations deployment. It allows you to quickly
test out configs for system components, especially Datasources,
Checkpoints, and Stores.</p>
<p>For many deployments of Great Expectations, these components (plus
Expectations) are the only ones you’ll need.</p>
<p>test_yaml_config is mainly intended for use within notebooks and tests.</p>
<p>–Public API–</p>
<dl class="option-list">
<dt><kbd><span class="option">--Documentation--</span></kbd></dt>
<dd><p><a class="reference external" href="https://docs.greatexpectations.io/docs/terms/data_context">https://docs.greatexpectations.io/docs/terms/data_context</a>
<a class="reference external" href="https://docs.greatexpectations.io/docs/guides/validation/checkpoints/how_to_configure_a_new_checkpoint_using_test_yaml_config">https://docs.greatexpectations.io/docs/guides/validation/checkpoints/how_to_configure_a_new_checkpoint_using_test_yaml_config</a></p>
</dd>
</dl>
<dl class="field-list simple">
<dt class="field-odd">Parameters</dt>
<dd class="field-odd"><ul class="simple">
<li><p><strong>yaml_config</strong> – A string containing the yaml config to be tested</p></li>
<li><p><strong>name</strong> – (Optional) A string containing the name of the component to instantiate</p></li>
<li><p><strong>pretty_print</strong> – Determines whether to print human-readable output</p></li>
<li><p><strong>return_mode</strong> – Determines what type of object test_yaml_config will return.
Valid modes are “instantiated_class” and “report_object”</p></li>
<li><p><strong>shorten_tracebacks</strong> – If true, catch any errors during instantiation and print only the
last element of the traceback stack. This can be helpful for
rapid iteration on configs in a notebook, because it can remove
the need to scroll up and down a lot.</p></li>
</ul>
</dd>
